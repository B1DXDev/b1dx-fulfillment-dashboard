แบบที่ 1 — Reverse Spec (ดูโค้ดที่มีอยู่ → เขียน spec)

ใช้เมื่อ: มีโค้ดอยู่แล้ว แต่ยังไม่มีเอกสาร

"อ่านโค้ด Role & RBAC ที่มีอยู่ใน be-repo และ web-repo
 แล้วเขียน spec ออกมาเป็น docs/specs/rbac.md

 ให้ครอบคลุม:
 - Data model ที่มีอยู่จริง
 - API endpoints ที่มีจริง (path, method, request, response)
 - Business rules ที่โค้ดบังคับอยู่
 - Validation rules

 อย่าสมมติ — ดูจากโค้ดจริงเท่านั้น
 แล้วสรุปให้ confirm ก่อน save"

ตัวอย่าง: Feature Roles & RBAC

1
สร้าง Feature Request ใน tasks/current-sprint.md 
# Feature: Roles & RBAC

## Goal
จัดการ Role และ Permission ให้ Admin กำหนดสิทธิ์ User ได้

## Actors
- Super Admin: สร้าง/แก้/ลบ Role, กำหนด Permission
- Admin: มอบหมาย Role ให้ User
- User: เห็นเฉพาะ menu/action ที่ตัวเองมีสิทธิ์

## Core Workflows
1. สร้าง/แก้ไข/ลบ Role
2. กำหนด Permission ให้ Role
3. มอบหมาย Role ให้ User
4. Guard route/component ตาม Permission

## Out of Scope (MVP)
- Row-level security
- Permission inheritance

เช่น Background Job, Webhook, Email Notification, Cron

กฎที่ต้องจำ — Red Flags ที่ BLOCK ทันที
🚨
11 Anti-Patterns นี้ = BLOCKER ทันที — ไม่มีข้อยกเว้น Reviewer จะ block PR เสมอ
#Anti-Patternปัญหาวิธีที่ถูก
RF-01 Query ไม่กรอง tenant_id ข้อมูล leak ข้าม tenant เพิ่ม .Where(o => o.TenantId == tenantId)
RF-02 TenantId จาก request body attacker ควบคุม tenant ได้ ใช้ ICurrentTenant.RequiredTenantId
RF-03 Auth check หลัง fetch data fetch ก่อน check permission check permission ก่อนเสมอ
RF-04 Secret ใน source code credential leaked ใน git ใช้ env var / Secret Manager
RF-05 Raw SQL string interpolation SQL Injection ใช้ parameterized query / LINQ
RF-06 <a href> ใน app route (FE) ล้าง access token จาก memory ใช้ router.push() หรือ <Link>
RF-07 Unbounded list query OOM บน large tenant data ใส่ pagination + Math.Clamp(pageSize,1,100)
RF-08 Broken build committed CI พัง ทีมทำงานไม่ได้ dotnet build && dotnet test ต้องผ่านก่อน push
RF-09 Spec contract พัง BE/FE ต่อกันไม่ได้ แก้ spec ก่อน แล้ว build ตาม spec ใหม่
RF-10 Cross-feature import (FE) ทำลาย feature isolation ย้ายไป shared/ ถ้าต้องใช้ร่วมกัน
RF-11 Token ใน localStorage (FE) XSS ขโมย token ได้ Access token: in-memory / Refresh token: httpOnly cookie

Path Rules & Submodule
📌
กฎทอง: Submodule ใน workspace = read-only spec mirror · เขียนโค้ดที่ {*_repo_path} เท่านั้น (ตาม workspace.local.json)
Submodule = read-only ref (pinned commit) · Direct repo = write-able working copy
📁 Path Rules
กฎเหตุผล
อ่าน path จาก workspace.local.jsonแต่ละเครื่องมี path ต่างกัน ห้าม hardcode
เขียนโค้ดใน {be_repo_path} / {fe_repo_path} / {sync_worker_repo_path} / {webhook_repo_path} เท่านั้นsubmodule เป็น read-only reference
ห้ามแตะ ./be-repo/, ./web-repo/, ./sync-worker-repo/, ./webhook-repo/ ในการเขียนโค้ดจะทำให้ submodule diverge จาก direct repo
เมื่อ submodule out of sync
# reset submodule กลับไป main
git -C be-repo           checkout main && git -C be-repo           pull origin main
git -C web-repo          checkout main && git -C web-repo          pull origin main
git -C sync-worker-repo  checkout main && git -C sync-worker-repo  pull origin main
git -C webhook-repo      checkout main && git -C webhook-repo      pull origin main

# ตรวจสอบ
git submodule status
🔀 Git Rules เพิ่มเติม
กฎเหตุผล
รัน git status --short ก่อน build ทุกครั้งป้องกันเขียนทับงานที่ค้างอยู่
ถ้ามี local changes → อ่านก่อน ไม่เขียนทับlocal changes มักสมบูรณ์กว่าที่ agent จะเขียนใหม่
ห้าม commit workspace.local.jsonมี path เฉพาะเครื่อง (gitignored)
Agent ห้าม auto commit / push / PRstage ได้ → รอ user สั่ง commit / push / PR
BLOCKING ห้าม push ตรงเข้า main ทุก repoทุก change ต้องผ่าน PR + reviewer merge · hard-stop ถ้า user พิมพ์ push main ตรง

Lessons Learned (เคสจริง)

ข้อผิดพลาดที่เคยเกิด + วิธีป้องกัน — เก็บไว้กันลืม

❌ #1 — Agent เขียนโค้ดลง submodule โดยตรง
สิ่งที่เกิด
Agent เขียนโค้ดลง ./be-repo/ → submodule diverge จาก direct repo → ต้องย้าย commit ด้วยมือทีหลัง 😓
วิธีป้องกัน
workspace.local.json ระบุ path จริง → Agent เขียนโค้ดลง {be_repo_path} ✅
❌ #2 — Agent เขียนทับ local changes
สิ่งที่เกิด
Dev มีงานค้างใน repo จริง → Agent ไม่รู้ → เขียนโค้ดใหม่ทับ → local changes หาย (มักสมบูรณ์กว่า) 😓
วิธีป้องกัน
ก่อน build → git status --short
มี changes → อ่านก่อน ไม่เขียนทับ ✅
ไม่มี → เขียนโค้ดใหม่ได้เลย ✅
❌ #3 — Hardcode path ใน CLAUDE.md
สิ่งที่เกิด
CLAUDE.md: be_repo = D:\2026\pa\... ← path ของคนเดียว → Dev คนอื่น clone → path ไม่ตรง → พัง 😓
วิธีป้องกัน
workspace.local.json (gitignored) — แต่ละคนใส่ path ของตัวเอง
CLAUDE.md อ้างอิง {be_repo_path} แทน ✅
❌ #4 — Commit/Push ตรงเข้า main (3 ครั้งในวันเดียว · 2026-05-13)
สิ่งที่เกิด
Agent run sequence:
git checkout main && git merge feature/x && git push origin main
→ ไม่มี PR, ไม่มี review, ไม่มี context 😓
วิธีป้องกัน
Hardened rule: agent ห้าม push main ทุก repo
→ push feature branch → gh pr create → reviewer merge
ถ้า user พิมพ์ push main → hard-stop + require explicit confirmation each time
❌ #5 — Visual fidelity miss (EOD-FE-04 · 2026-04-27)
สิ่งที่เกิด
PR merged แล้วแต่ visual ขาด accent bar, header icon+pill, audit trail rows
Tests ผ่านครบ + data-testids ครบ → reviewer approve → user เปิดมาเห็นไม่ตรง mockup 😓
วิธีป้องกัน
Mockup Visual Fidelity (BLOCKING):
• FE Coder: dev server → screenshot vs mockup → checklist 7 ข้อ
• Reviewer: ห้าม approve ก่อนเปิด mockup จริง → require PR screenshot
🎯
Lesson: tests passing ≠ feature เสร็จ — visual fidelity เป็น acceptance criteria เสมอ

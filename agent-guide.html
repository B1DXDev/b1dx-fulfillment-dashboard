1. ภาพรวม Agent Architecture

Claude Code ทำงานแบบ Orchestrator + Subagents — Orchestrator คุยกับ user, ส่งงานให้ specialized subagents แล้วรวบรวมผลลัพธ์กลับมา

Pipeline overview — Orchestrator แตกงานเป็น parallel subagents แล้วรวมผลที่ Phase 3
📋 When to spawn each subagent
SubagentSpawn เมื่อWorking dirReference
🧪 UI Tester ×2 ทุก feature ที่มี FE — Phase 1.5 (scaffold) + Phase 3 (run) {fe_repo_path} ui-tester.md
⚙️ BE Subagent API / database / business logic ฝั่ง backend {be_repo_path} be-repo/CLAUDE.md
🎨 FE Subagent UI / page / hook / component {fe_repo_path} web-repo/CLAUDE.md
🔄 Sync Worker ERP → OMS sync (Checkpoint / Staging / Outbox) {sync_worker_repo_path} sync-worker-coder.md
🪝 Webhook Coder Marketplace webhook (Shopee/Lazada/TikTok/Facebook/LINE), OAuth, RabbitMQ DLX/DLQ, deploy ops {webhook_repo_path} webhook-coder.md + trigger flow
🔍 Reviewer auth/JWT/tenant isolation, shared packages, business-critical flow, dev ร้องขอตรงๆ any reviewer.md

# Full pipeline (Orchestrator spawn ครบทุก phase + agent)

# สร้าง feature ใหม่ (ระบุชื่อ + description สั้นๆ)
claude "อ่าน agents/orchestrator.md แล้ว build feature order-tracking: real-time order status"

# ทำแค่ฝั่งเดียว
claude "build BE for order-tracking — อ่าน be-repo/CLAUDE.md + docs/specs/order-tracking.md"
claude "build FE for order-tracking — อ่าน web-repo/CLAUDE.md + docs/specs/order-tracking.md"

# ต่อ session เดิม / debug
claude --continue
claude --verbose

# ไม่ถาม confirm (ใช้เมื่อแน่ใจใน spec)
claude --dangerously-skip-permissions "..."

claude "handoff S2-BE-01 from nuchit to wat"             # โอน owner
claude "suggest parallel task: role=fe — waiting for S2-BE-01"

4. Working Protocol (3 Phases)

Claude จะทำตาม 3 phases นี้เสมอ — ห้าม skip

3-Phase pipeline พร้อม estimated timing — phase 2 ทำ parallel ระหว่าง BE และ FE
Phase 1 🎨 Design
ขั้นสิ่งที่ทำOutput
1อ่าน be-repo/CLAUDE.md + web-repo/CLAUDE.mdcontext loaded
2อ่าน mockup ที่ spec ระบุ (fallback docs/mockups/b1dx_full_app_v0.html)UI source of truth
3ออกแบบ data model + API contractER diagram + endpoints
4เขียน spec → docs/specs/{feature}.mdspec.md
5แสดง test plan + รอ user confirm ⏸confirmed spec
Phase 1.5 🧪 Scaffold Tests

UI Tester Subagent ทำงาน:

InputProcessOutput
spec.mdacceptance criteria → test casese2e/{feature}/{feature}.spec.ts
feature → happy + edgetest.skip ทุกตัว (intentional)+ .po.ts ถ้าซับซ้อน
bug → regression reproducebranch: feat/{branch} หรือ fix/{branch}commit: test(e2e): scaffold
improve → before/after(เดียวกับ FE Coder)—
Phase 2 ⚡ Build (parallel)

ทุก subagent ต้อง:

  1. git status --short ก่อนเริ่ม
  2. git checkout main && git pull && git checkout -b {branch}
  3. TDD — test ก่อน impl ทุก layer
  4. หลัง push → node scripts/gen-progress.mjs
🐛
Bug fix: เขียน failing regression test ที่ reproduce bug ก่อนเสมอ

5. กฎที่ Claude ต้องปฏิบัติ
5.1 📁 Path Rules
กฎเหตุผล
อ่าน path จาก workspace.local.jsonแต่ละเครื่องมี path ต่างกัน ห้าม hardcode
เขียนโค้ดใน {be_repo_path} / {fe_repo_path} / {sync_worker_repo_path} / {webhook_repo_path} เท่านั้นsubmodule เป็น read-only reference
ห้ามแตะ ./be-repo/, ./web-repo/, ./sync-worker-repo/, ./webhook-repo/ ในการเขียนโค้ดจะทำให้ submodule diverge จาก direct repo
5.2 🔀 Git Rules
กฎเหตุผล
รัน git status --short ก่อน build ทุกครั้งป้องกันเขียนทับงานที่ค้างอยู่
ถ้ามี local changes → อ่านก่อน ไม่เขียนทับlocal changes มักสมบูรณ์กว่าที่ agent จะเขียนใหม่
แตก branch ก่อน commit เสมอ — ห้าม commit ตรงเข้า mainไม่มี PR, ไม่มี review, ไม่มี context
BLOCKING ห้าม push ตรงเข้า mainทุก change ต้องผ่าน PR + reviewer merge
ห้าม commit workspace.local.jsonมี path เฉพาะเครื่อง
Agent ห้าม auto commit/push/PRstage ได้ → รอ user สั่ง "commit"/"push"/"PR"
5.3 💻 Code Rules
กฎใช้กับ
ห้ามรับ tenant_id จาก request body — ใช้ ICurrentTenantBE
ห้าม deep import @b1dx/ui/src/...FE
ห้ามใช้ <a href> ใน (app) routes — ใช้ router.push()FE
user-visible text ต้องใช้ t('key') ทุกตัวFE
Spec คือ source of truth — ห้ามแก้ระหว่าง buildALL
ทุก SQL ผ่าน Dapper parameter — ห้าม string interpolationSync
Upsert ต้อง idempotent (ON CONFLICT DO UPDATE) + checkpoint update ครบทุก pathSync
Webhook handler return 202 + publish RabbitMQ — ห้าม process synchronousWebhook
Upsert ใช้ (platform, seller_account, external_id) ON CONFLICT — tenant_id ใน DoUpdatesWebhook
OAuth callback verify state HMAC + 409 on cross-tenant bindWebhook
ใช้ constructor injection ใน cmd/api/main.go — ห้าม global / DI frameworkWebhook
Logging stdlib log + prefix [Platform]/[Worker]/[Consumer]Webhook

6. Submodule vs Direct Repo
Submodule = read-only ref (pinned commit); Direct repo = write-able working copy
📌
กฎทอง: Submodule ถูก pin ไว้ที่ commit เฉพาะ — ไม่ auto-update ตาม direct repo
เขียนโค้ดที่ {*_repo_path} เท่านั้น
6.1 เมื่อ submodule out of sync
# reset submodule กลับไป main
git -C be-repo           checkout main && git -C be-repo           pull origin main
git -C web-repo          checkout main && git -C web-repo          pull origin main
git -C sync-worker-repo  checkout main && git -C sync-worker-repo  pull origin main
git -C webhook-repo      checkout main && git -C webhook-repo      pull origin main

# ตรวจสอบ
git submodule status

7. Spec Management

Spec ทุกอย่างอยู่ที่ docs/specs/ — BE และ FE อ่านจากที่เดียวกัน

📂 Spec Directory
docs/specs/
├── auth.md              ← auth flow, JWT strategy
├── orders-fetch-hooks.md
└── {feature-name}.md    ← สร้างใหม่ก่อน build ทุก feature
Spec Lifecycle — Draft → Confirm → Build (with feedback loop ถ้าเจอ edge case)
📌
กฎทอง: Spec มีที่เดียวเท่านั้น — docs/specs/ ห้าม copy ไปไว้ที่อื่น
หากต้องแก้ spec กลางทาง → แก้ที่ docs/specs/ + แจ้ง user ก่อน + update โค้ดที่ build ตามด้วย

8. เมื่อไหรต้องเขียน Spec
✅ ต้องเขียน Spec เมื่อ
  • Feature ใหม่ — data model / endpoint / page ใหม่
  • BE + FE ต้องทำพร้อมกัน — ต้องตกลง contract
  • Design decision ใหญ่ — auth flow, permission model
❌ ไม่ต้องเขียน Spec เมื่อ
  • Bug fix — แค่บอกอาการ + expected
  • UI improve — z-index, component กลาง, style
  • Refactor — ที่ไม่เปลี่ยน behavior
  • งาน FE-only ที่ไม่ต้องการ API ใหม่
📊 ตัวอย่าง
งานSpec?เหตุผล
เพิ่ม country code picker ใน phone field❌FE-only, บอก intent ใน chat พอ
สร้าง RHF wrapper component กลาง❌ไม่มี API เกี่ยวข้อง
แก้ z-index Combobox ใน drawer❌bug fix
สร้าง feature User Management ทั้ง BE+FE✅ต้องตกลง contract
Auth flow + JWT✅design decision ใหญ่
Migration table ใหม่ + endpoint ใหม่✅data model change
Refactor query เป็น CQRS — output เท่าเดิม❌behavior identical
💡
สรุป: ถ้าต้องตกลงกันว่า API หน้าตาเป็นยังไง → เขียน spec
ถ้าทำได้เลยโดยไม่ต้องตกลงอะไร → บอกใน chat พอ
⚡ สำหรับงาน improve/fix ที่ชัดเจน — สั่งใน chat ได้เลย
claude "fix: UserDetailDrawer phone field ให้ใช้ RHFPhoneInput แทน RHFTextInput"
claude "improve: เพิ่ม loading skeleton ให้ DataTable"

10. Lessons Learned

ข้อผิดพลาดที่เคยเกิด + วิธีป้องกัน

❌ #1 — Agent เขียนโค้ดลง submodule โดยตรง
สิ่งที่เกิด
Agent เขียนโค้ดลง ./be-repo/ → submodule diverge จาก direct repo → ต้องย้าย commit ด้วยมือทีหลัง 😓
วิธีป้องกัน
workspace.local.json ระบุ path จริง → Agent เขียนโค้ดลง {be_repo_path} ✅
❌ #2 — Agent เขียนทับ local changes
สิ่งที่เกิด
Dev มีงานค้างใน repo จริง → Agent ไม่รู้ → เขียนโค้ดใหม่ทับ → local changes หาย (มักสมบูรณ์กว่า) 😓
วิธีป้องกัน
ก่อน build → git status --short
มี changes → อ่านก่อน ไม่เขียนทับ ✅
ไม่มี → เขียนโค้ดใหม่ได้เลย ✅
❌ #3 — Hardcode path ใน CLAUDE.md
สิ่งที่เกิด
CLAUDE.md: be_repo = D:\2026\pa\... ← path ของคนเดียว → Dev คนอื่น clone → path ไม่ตรง → พัง 😓
วิธีป้องกัน
workspace.local.json (gitignored) — แต่ละคนใส่ path ของตัวเอง
CLAUDE.md อ้างอิง {be_repo_path} แทน ✅
❌ #4 — Commit/Push ตรงเข้า main (3 ครั้งในวันเดียว — 2026-05-13)
สิ่งที่เกิด
Agent run sequence:
git checkout main && git merge feature/x && git push origin main
→ ไม่มี PR, ไม่มี review, ไม่มี context 😓
วิธีป้องกัน
Hardened rule: agent ห้าม push main ทุก repo
→ push feature branch → gh pr create → reviewer merge
ถ้า user พิมพ์ push main → hard-stop + require explicit confirmation each time
❌ #5 — Visual fidelity miss (EOD-FE-04, 2026-04-27)
สิ่งที่เกิด
PR merged แล้วแต่ visual ขาด accent bar, header icon+pill, audit trail rows
Tests ผ่านครบ + data-testids ครบ → reviewer approve → user เปิดมาเห็นไม่ตรง mockup 😓
วิธีป้องกัน
Mockup Visual Fidelity (BLOCKING):
• FE Coder: dev server → screenshot vs mockup → checklist 7 ข้อ
• Reviewer: ห้าม approve ก่อนเปิด mockup จริง → require PR screenshot
🎯
Lesson: tests passing ≠ feature เสร็จ — visual fidelity เป็น acceptance criteria เสมอ

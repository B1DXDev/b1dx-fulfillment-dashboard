🪝 เมื่อไหร่ที่ Webhook Coder ถูก spawn?

Webhook Coder เป็น specialist ใหม่ใน Phase 2 — Orchestrator จะ spawn เฉพาะเมื่อ feature เข้าเงื่อนไขใดเงื่อนไขหนึ่งใน 4 categories นี้ (รายละเอียดเงื่อนไขครบ: docs/guides/webhook-coder-trigger-flow.md)

📨 Webhook Ingestion รับ event จาก marketplace (Shopee / Lazada / TikTok / Facebook / LINE) — HTTP 202 ทันที + publish RabbitMQ
🔐 OAuth Platform Bind เชื่อม merchant account ↔ marketplace platform · token refresh · scope management
📬 RabbitMQ DLX-DLQ Dead-letter exchange + queue · retry policy · idempotency key per event
🛒 Marketplace Integration Platform-specific adapter · signature verification · webhook → order sync

⚠️ Webhook handler ต้อง return HTTP 202 ทันที + publish RabbitMQ — ห้าม process sync · ทุก Upsert ต้อง idempotent

  • UI Tester un-skip + run E2E

⚖️ Single Agent vs Agent Team

❌ Single Agent (แบบเก่า)

  • 1 context ทำทุกอย่างเอง
  • System prompt เดียว ครอบคลุมทุกเรื่อง
  • ทำงานแบบ sequential ทีละขั้น
  • Context พองเร็ว — สูญเสีย accuracy
  • ไม่มี separation of concerns
  • ไม่มี quality gate ที่ enforce ได้

✅ Agent Team (Solution นี้)

  • Orchestrator + spawn specialist subagents
  • 10 system prompts เฉพาะแต่ละ role
  • Parallel — BE+FE+Sync+Webhook ทำพร้อมกัน
  • Context แต่ละ subagent แยก ไม่ปนกัน
  • BE / FE / Sync / Webhook / Test / Review แยกชัด
  • Reviewer + UI Tester เป็น quality gate

🚦 Pre-Build Plan Gate & Auto-Proceed Stop List

ก่อนแก้โค้ดแม้แต่บรรทัดเดียว — agent ต้องแสดง plan 4 รายการ → สร้าง tasks/plans/{TICKET-ID}/plan.md → รอ user trigger (build / ลุย) → build ต่อเนื่องโดย ไม่ confirm กลางทาง ยกเว้นชน 13 เงื่อนไขด้านล่าง

📋 Plan Gate: 4 รายการที่ agent ต้องแสดงก่อน build
  1. Plan & Scope — ตารางไฟล์ที่แก้/เพิ่ม + สิ่งที่เปลี่ยน + out of scope
  2. Summary of changes — ตาราง current → expected ต่อ item
  3. SIT Impact — flow/feature อื่นที่อาจได้รับผลกระทบ (ระบุถ้าไม่มีด้วย)
  4. Test Plan — unit ทุก layer + E2E (happy + edge + error)
🛑 Auto-Proceed Stop List — 13 เงื่อนไขที่ agent ต้องหยุดถาม user
TIER 1 MUST stop — irreversible / data risk / cross-team
  1. งานไม่ตรง spec / spec ขัด mockup
  2. Scope change (เพิ่ม/ลด feature นอก plan)
  3. Blocker: missing dep, BE contract ไม่พร้อม, conflict งาน in-flight
  4. Irreversible: drop table, force-push, delete branch, migration backfill+lock
  5. Security: credential exposure, auth bypass, tenant isolation leak
  6. Breaking change: remove/rename public API, shared contract field
  7. Cross-repo impact: @b1dx/ui, shared types, infra/CI/CD, env vars
  8. External cost: paid API, LLM call ใน build, dep license unclear
  9. Commit / Push / PR — stage + รอ user สั่ง
TIER 2 SHOULD stop — drift จาก plan
  1. Effort blow-up >2x จาก plan
  2. Hidden bug นอก scope → log backlog.md แล้วถาม fix ตอนนี้ไหม
  3. Test เดิม fail หลัง change · root cause unclear
  4. Business rule ambiguous: rounding / timezone / permission / mockup copy
TIER 3 INFORM ใน report — ไม่ block
  • Flaky test (mark + แจ้ง)
  • Dead code ลบเพื่อ proceed (แจ้งใน diff)
  • Dep ใหม่ size ปกติ (<1MB)

💰 Token Efficiency Rules

ทำงานให้เสร็จด้วย token น้อยที่สุด — ไม่ใช่ทำงานน้อยที่สุด · RTK proxy ช่วยตัด output 60-90%

📖 Read & Search — ใช้ tool ที่ถูก
  • ห้าม Read ไฟล์เต็มถ้ารู้ว่าต้องการแค่บางส่วน → ใช้ offset + limit
  • ห้าม cat / head / tail — ใช้ Read tool เสมอ
  • ห้าม find / ls -R — ใช้ Glob
  • ห้าม grep / rg ใน Bash — ใช้ Grep tool
  • Search ก่อน Read — Grep หาตำแหน่ง → Read เฉพาะ offset/limit
  • ไฟล์ใหญ่ (>500 บรรทัด) เช่น tasks/backlog.md — ห้าม Read เต็ม
🤖 Subagent Delegation — ประหยัด context หลัก
  • งาน research/exploration ข้าม >3 ไฟล์ → spawn Explore / general-purpose
  • งาน output ยาว (test results, build output, log) → delegate ให้ subagent สรุป
  • ห้าม spawn subagent ซ้ำซ้อน — ใช้ SendMessage ต่อ agent เดิมแทน
  • Subagent prompt ต้อง self-contained — ระบุไฟล์ + บรรทัดที่ต้องอ่าน
🔧 Tool Selection & Editing
  • ใช้ dedicated tool เสมอ — Read/Edit/Write/Glob/Grep ก่อน Bash
  • Parallel tool calls — independent calls อยู่ใน message เดียว (เช่น git status + git diff)
  • ใช้ Edit ไม่ใช่ Write สำหรับไฟล์ที่มีอยู่ — ส่งเฉพาะ diff
  • replace_all: true ถ้า rename across file
  • ห้าม Read ก่อน Edit ถ้าเพิ่ง Write/Read ไปแล้วใน turn เดียวกัน
💬 Communication & File Discipline
  • คำตอบสั้น — ห้าม summarize งานตอนจบทุกครั้ง (user เห็น diff อยู่แล้ว)
  • ห้าม narrate ทุกขั้นตอน — บอกแค่เริ่ม + เปลี่ยนทิศทาง + จุดที่ติด
  • ห้าม preamble / postamble — "ฉันจะ..." / "หวังว่า..." ตัดออก
  • ภาษาตามที่ user ใช้ — ไทยตอบไทย, อังกฤษตอบอังกฤษ
  • ห้ามสร้างไฟล์ใหม่ ถ้าแก้ของเดิมได้
  • Plan file → tasks/plans/{TICKET-ID}/plan.md เท่านั้น

🎁 Business Benefits

⚡ Speed
BE + FE + Sync + Webhook ทำงาน parallel — ลดเวลา delivery feature 2-4 เท่า เทียบกับ sequential
🎯 Quality
UI Tester + Reviewer เป็น mandatory gate — ป้องกัน visual regression และ code quality issue
📊 Traceability
ทุก task มี plan.md + lifecycle status (Todo→Active→In Review→Done) — ติดตามได้ทุกขั้นตอน
🔒 Consistency
แต่ละ specialist มี CLAUDE.md + system prompt — บังคับ pattern เดียวกันทุกครั้ง
💰 Cost Efficiency
Context isolation + RTK token optimizer — ประหยัด token 60-90% ต่อ operation
📈 Scalability
เพิ่ม agent ใหม่ได้ง่าย (Webhook Coder เพิ่งเข้าทีม) โดยไม่กระทบ agent เดิม

💬 รูปแบบการสื่อสาร — Hierarchical (ไม่ใช่ Peer-to-Peer)

รูปแบบSolution นี้หมายเหตุ
Hierarchical (Coordinator → Workers) ✓ ใช้ Orchestrator spawn subagent → return ผลกลับ
Peer-to-Peer (Agents คุยกันเอง) ✗ ไม่ใช้ ไม่มี agent คุยกันโดยตรง
Shared File State ✓ ใช้ spec.md, plan.md, backlog.md, assignments/
Context Isolation ✓ ใช้ subagent return summary ไม่ pollute main context

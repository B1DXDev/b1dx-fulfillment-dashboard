UUID / Primary Key / Foreign Key — เลือกใช้อย่างไร

ปัญหาของ UUID v4 เป็น PK

UUID v4 สุ่มแบบ random ทำให้ B-Tree index แตกกระจาย (fragmented) เพราะทุก insert กระจายทั่ว index แทนที่จะ append ต่อท้าย ส่งผลให้ page splits บ่อย = query ช้า + insert ช้า

Type ขนาด Insert Speed Index Locality Global Unique ใช้เมื่อ
bigserial 8B เร็วสุด Sequential Single DB Default สำหรับทุก table
uuid v7 16B ดี Time-ordered ใช่ Multi-DB / API exposed / Distributed
uuid v4 16B ช้า Random ใช่ หลีกเลี่ยงเป็น PK table ใหญ่

Pattern ที่แนะนำ — Dual Key

CREATE TABLE orders (
    id          bigserial    PRIMARY KEY,       -- internal FK (8B, sequential, เร็ว)
    public_id   uuid         NOT NULL DEFAULT gen_random_uuid(),  -- external/API
    UNIQUE (public_id)
);

-- API/URL ใช้ public_id  → ป้องกัน enumeration
-- JOIN ภายในใช้ id       → bigint เร็วกว่า uuid 2x
💡
กฎง่าย ๆ FK ทุก column ต้องมี Index แยกเสมอ — PostgreSQL ไม่ auto-create ให้เหมือน MySQL

ปัญหาของการยัดทุกอย่างไว้ใน Record เดียว

สมมติตาราง orders เก็บทุกอย่างรวมกัน:

| order_id | customer_name | customer_email | product_name | price | category | warehouse |
|----------|---------------|----------------|--------------|-------|----------|-----------|
| 1        | สมชาย         | som@email.com  | iPhone 15    | 35000 | มือถือ   | กรุงเทพ   |
| 2        | สมชาย         | som@email.com  | iPhone 15    | 35000 | มือถือ   | กรุงเทพ   |
| 3        | สมชาย         | som@email.com  | Samsung S24  | 28000 | มือถือ   | กรุงเทพ   |
ปัญหาอธิบายผลกระทบ
Data Redundancy ข้อมูลลูกค้า / สินค้าซ้ำทุก record Storage โตเร็วโดยไม่จำเป็น
Update Anomaly แก้ email ลูกค้า → ต้องแก้ทุก row ถ้า crash กลางทาง = ข้อมูลไม่ consistent
Insert Anomaly อยากเพิ่มสินค้าใหม่ แต่ยังไม่มี order ทำไม่ได้ — ต้องมี order ก่อน
Delete Anomaly ลบ order ที่มีสินค้าเดียว ข้อมูลสินค้านั้นหายไปจาก DB
Schema บวม GROUP BY string อาจมี typo "มือถือ" vs "มือถือ " = 2 กลุ่ม รายงานผิด

Normalization คืออะไร ทำไมต้องทำ

Normalization คือการจัดเก็บข้อมูลแต่ละชนิดไว้ใน "ที่เดียว" และให้ตารางอื่น อ้างอิง ผ่าน Foreign Key แทนการคัดลอก

Normal Formกฎตัวอย่างที่ fix
1NF ทุก cell มีค่าเดียว — ไม่มี array ใน column tags = "A,B,C" → แยก table product_tags
2NF ทุก column ขึ้นกับ PK ทั้งหมด product_name ใน order_items → ย้ายไป products
3NF ไม่มี column ขึ้นกับ column อื่น (non-PK) category_name ขึ้นกับ category_id → แยก categories

❌ ไม่ Normalize

  • แก้ที่เดียว ไม่ถูกทุกที่
  • ข้อมูลซ้ำ = storage waste
  • รายงานอาจผิด
  • Schema แก้ยาก

✅ Normalize

  • แก้ที่เดียว → ถูกทุกที่ทันที
  • FK constraint รับประกัน integrity
  • Storage ประหยัด 40–60%
  • Schema ยืดหยุ่นรองรับการเติบโต

ทำไมต้องแยก Product / Images / Prices

⚠️
กฎง่าย ๆ — ถาม "1 สินค้า มีได้กี่ ___?" ถ้าคำตอบคือ "หลาย" → ต้องแยกตารางเสมอ
สิ่งที่เก็บจำนวนต่อ 1 สินค้าเก็บที่ไหน
ชื่อสินค้า / SKU1column ใน products
รูปภาพหลายรูป + มี sortแยก product_images
ราคาหลาย tier + มีวันหมดอายุแยก product_prices
Variant (size/color)หลาย variantแยก product_variants
Tag / Categoryหลาย tagแยก product_tags

โครงสร้างที่ถูกต้อง

CREATE TABLE products (
    id          bigserial PRIMARY KEY,
    public_id   uuid UNIQUE DEFAULT gen_random_uuid(),
    name        text NOT NULL,
    sku         text UNIQUE NOT NULL
);

-- 1 สินค้า → หลายรูป, มี sort_order
CREATE TABLE product_images (
    id          bigserial PRIMARY KEY,
    product_id  bigint NOT NULL REFERENCES products(id),
    url         text NOT NULL,
    sort_order  int NOT NULL DEFAULT 0,
    is_primary  boolean NOT NULL DEFAULT false
);

-- 1 สินค้า → หลาย tier, มีวันหมดอายุ
CREATE TABLE product_prices (
    id          bigserial PRIMARY KEY,
    product_id  bigint NOT NULL REFERENCES products(id),
    price_type  text NOT NULL,  -- 'retail' | 'wholesale' | 'member' | 'promo'
    amount      numeric(12,2) NOT NULL,
    valid_from  timestamptz,
    valid_until timestamptz
);
สถานการณ์❌ ไม่แยก✅ แยกตาราง
สินค้ามี 10 รูปต้องเพิ่ม columnINSERT เพิ่มแถวได้เลย
เรียงรูปใหม่ทำไม่ได้UPDATE sort_order
Flash Sale มีวันหมดอายุทำไม่ได้ใส่ valid_until
ราคาต่างสาขาต้อง ALTER TABLEเพิ่ม branch_id
ลบรูปที่ 3SET NULLDELETE แถวนั้น

อธิบายให้ผู้บริหารเข้าใจ

🏢
เปรียบเหมือนการจัดการแฟ้มเอกสารในสำนักงาน
แทนที่จะเขียนชื่อ-ที่อยู่ลูกค้าซ้ำในทุกใบสั่งซื้อ ให้เก็บลูกค้าในสมุดลูกค้าเล่มเดียว แล้วให้ใบสั่งซื้ออ้างอิงรหัสลูกค้าแทน — ลูกค้าย้ายที่อยู่ แก้ที่เดียว ถูกทุกใบทันที

ผลกระทบทางธุรกิจ

ด้านไม่แยกข้อมูลแยกข้อมูล
รายงาน ตัวเลขไม่ตรงกันระหว่างรายงาน — ผู้บริหารได้ข้อมูลคนละชุดในการประชุมเดียวกัน ข้อมูล consistent เสมอ เชื่อถือได้
การเงิน ราคาในระบบกับราคาจริงไม่ตรง → เรียกเก็บเงินผิด / คืนเงินผิด แก้ราคาที่เดียว → ถูกทุก order
Compliance ข้อมูลลูกค้าไม่ consistent → ปัญหา PDPA จัดการ data lifecycle ได้ถูกต้อง
Scale ระบบช้าลงเรื่อย ๆ ตามปริมาณข้อมูล แก้ยากมากเมื่อข้อมูลโตแล้ว รองรับข้อมูลหลักล้านได้โดยไม่ช้า
ค่าใช้จ่าย Storage สูงจากข้อมูลซ้ำ + ค่าแก้ไขข้อมูลในอนาคตสูงมาก ประหยัด storage 40–60% + ค่า maintenance ต่ำ 
ตัวอย่างตัวเลข:

สินค้า 1,000 รายการ · ขายได้ 100,000 orders

❌ ไม่แยก: ชื่อสินค้า + ราคา ถูกเก็บซ้ำ 100,000 ครั้ง
✅ แยก:    เก็บสินค้า 1,000 แถว, orders อ้างอิง product_id

→ ประหยัด storage ~90%
→ แก้ราคา: 1 คำสั่ง แทน 100,000 คำสั่ง
→ โอกาสข้อมูลผิดพลาด: ลดลงเกือบเป็นศูนย์

สรุปสำหรับผู้บริหาร

การแยกข้อมูลคือการลงทุนด้านความน่าเชื่อถือ — เหมือนการแยกแผนกในบริษัท ฝ่ายบัญชีไม่คัดลอกข้อมูลลูกค้าไว้เอง แต่อ้างอิงจากฝ่ายขายที่เดียว ทำให้ข้อมูลทั้งบริษัทตรงกันเสมอ

ต้นทุน: เวลาออกแบบเพิ่ม 20–30% ในช่วงแรก  |  ผลตอบแทน: ระบบถูกต้อง เชื่อถือได้ รองรับการเติบโต ลดค่าแก้ไขในอนาคต

Normalize ที่ Source, Denormalize ที่ Read Layer

🍽️
เปรียบเหมือนร้านอาหาร — ครัวเก็บวัตถุดิบแยกกัน (เนื้อ / หอม / กระเพรา / ข้าว) แต่จัดจานให้ลูกค้าเห็นครบในที่เดียว ครัวไม่ผสมทุกอย่างไว้ในถุงเดียวล่วงหน้า — แต่จัดจานเมื่อลูกค้าสั่ง

3 เครื่องมือสำหรับ Read Layer

1. Materialized View — ง่ายสุด เริ่มต้นที่นี่

CREATE MATERIALIZED VIEW order_summary AS
SELECT
    o.id, o.order_no,
    c.name     AS customer_name,
    o.total, o.status, o.created_at
FROM orders o
JOIN customers c ON c.id = o.customer_id;

-- Query รายงาน → ไม่มี JOIN แล้ว
SELECT * FROM order_summary WHERE status = 'pending';

-- Refresh ตามกำหนด
REFRESH MATERIALIZED VIEW CONCURRENTLY order_summary;

ผล: query ที่เคยใช้ 800ms → เหลือ 12ms

2. Read Replica — สำหรับ traffic สูง

App → [WRITE] → Primary DB   ← แก้ข้อมูลที่นี่ (source of truth)
                     │ sync อัตโนมัติ
                     ↓
      [READ]  → Replica DB   ← รายงาน / Dashboard ดึงจากที่นี่

ผลดี: Primary ไม่ช้าเพราะรายงาน
      Replica crash → Write ยังทำงานได้

3. Cache (Redis) — สำหรับข้อมูลที่ไม่เปลี่ยนบ่อย

App → ถามใน Cache ก่อน
         ├── มี  → ส่งกลับทันที (< 1ms)
         └── ไม่มี → ดึงจาก DB → เก็บ Cache → ส่งกลับ

ภาพรวม

WRITE PATH — Normalized (Source of Truth)
products / orders / customers — แยกตาราง / มี FK / Consistent เสมอ
↓ sync / refresh
READ PATH — Denormalized
Materialized View / Cache / Read Replica — เร็ว / ไม่มี JOIN / ไม่กระทบ Write

Architecture สำหรับหลักร้อยล้าน Record

ปัญหาหลักคือการใช้ DB เดียวทำทั้ง Transaction และ Report — ต้องแยก 2 โลกออกจากกัน

OLTP (Transaction)OLAP (Analytics)
จุดประสงค์เขียน / อ่านรายการวิเคราะห์ / รายงาน / BI
ข้อมูลปัจจุบันสะสมทุก period
Query patternแถวเดียว เร็วหลายล้านแถว + aggregate
Storage formatRow-basedColumn-based
เครื่องมือPostgreSQL / MySQLClickHouse / BigQuery / Redshift

Architecture 4 Layer

Layer 1 — OLTP (รับ Transaction)
Primary DB · Sharded by tenant/region · Partition by created_at · Index ครบทุก FK
↓ CDC (Change Data Capture) — Debezium / Kafka
Layer 2 — Pipeline (ลำเลียงข้อมูล)
Event Stream → Kafka → ETL/ELT → Transform · Batch: ทุกคืน / Stream: real-time <1 นาที
↓
Layer 3 — Data Warehouse (วิเคราะห์)
ClickHouse / BigQuery / Redshift · Columnar storage · Star Schema · Pre-aggregated
↓
Layer 4 — Serving (แสดงผล)
Dashboard → Cache (Redis) → API → Data Warehouse · Export → S3 / Object Storage

ทำไม Columnar Storage เร็วกว่า

❌ Row-based (PostgreSQL)

[id|amount|status|date]
[1 |35000 |done  |Jun1]
[2 |28000 |done  |Jun1]

SUM(amount) → อ่านทุก field
= 100GB I/O จาก 500M rows

✅ Column-based (ClickHouse)

[amount]: 35000,28000,15000...

SUM(amount) → อ่านแค่คอลัมน์นี้
= 4GB I/O จาก 500M rows

Star Schema — โครงสร้างใน Data Warehouse

                    fact_orders           ← ตารางหลัก (ร้อยล้านแถว)
                    ├── customer_key ─→ dim_customer  (ชื่อ / segment / region)
                    ├── product_key  ─→ dim_product   (ชื่อ / category / brand)
                    ├── date_key     ─→ dim_date       (year / month / quarter)
                    ├── amount
                    └── quantity

JOIN น้อยกว่า Normalized → aggregate เร็วกว่ามาก

ผลลัพธ์ที่ได้

รายงานก่อน (DB เดียว)หลัง (แยก Layer)
ยอดขายวันนี้45 วินาที< 1 วินาที
สรุปรายเดือน3 ชั่วโมง2 วินาที
Top 100 สินค้า20 นาที< 1 วินาที
Export CSVTimeout30 วินาที
กระทบ Productionใช่ (ระบบช้า)ไม่กระทบเลย

Partitioning — เพิ่มทีหลังได้ไหม

✅
ได้ — แต่ PostgreSQL ไม่มี ALTER TABLE ... PARTITION BY ต้องทำผ่าน migration

วิธีที่แนะนำ — Zero-Downtime Migration

-- 1. สร้าง table ใหม่ที่ partition แล้ว
CREATE TABLE orders_partitioned (
    LIKE orders INCLUDING ALL
) PARTITION BY RANGE (created_at);

-- 2. สร้าง partitions
CREATE TABLE orders_2025 PARTITION OF orders_partitioned
    FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');
CREATE TABLE orders_2026 PARTITION OF orders_partitioned
    FOR VALUES FROM ('2026-01-01') TO ('2027-01-01');

-- 3. Copy ข้อมูล (batch ทีละ range — ไม่ lock นาน)
INSERT INTO orders_partitioned
SELECT * FROM orders WHERE created_at >= '2025-01-01' AND created_at < '2026-01-01';

-- 4. Rename swap (atomic)
BEGIN;
  ALTER TABLE orders RENAME TO orders_old;
  ALTER TABLE orders_partitioned RENAME TO orders;
COMMIT;

-- 5. Drop เมื่อมั่นใจ
DROP TABLE orders_old;

ควรเพิ่ม Partitioning เมื่อไหร่

Record CountสัญญาณAction
< 1MQuery ยังเร็วยังไม่ต้อง — index พอ
1M – 5MReport เริ่มช้าเพิ่ม indexes + Materialized View ก่อน
5M – 10MEXPLAIN ยัง Seq Scanวางแผน Partition (ทำได้ใน maintenance window)
10M+ช้าจนรับไม่ได้เริ่มทำทันที

Roadmap การวางแผน (Practical)

Phase 1 · <10M Materialized View
+ Read Replica

ราคา: ถูก
Phase 2 · 10M–100M แยก Reporting DB
+ ETL Batch

ราคา: ปานกลาง
Phase 3 · 100M+ Data Warehouse
+ CDC Stream

ราคา: สูง แต่จำเป็น
💡
เคล็ดลับ — ออกแบบ created_at NOT NULL ไว้ตั้งแต่แรก เพราะ partition key ต้องไม่เป็น NULL — ทำให้ migration ทีหลังง่ายขึ้นมาก

สรุปหลักการทั้งหมด

Normalize ที่ source เพื่อให้ข้อมูลถูกต้อง ไม่ซ้ำซ้อน แก้ที่เดียวถูกทุกที่  ·  Denormalize ที่ read layer เพื่อ report เร็ว ไม่กระทบ production  ·  Partition ตาม scale เพิ่มทีหลังได้แต่วางแผนไว้ก่อนดีกว่า  ·  ยิ่งระบบโตต้องแยก OLTP ออกจาก OLAP — ความเร็วรายงานไม่ได้มาจาก optimize query แต่มาจากการเตรียมข้อมูลไว้ล่วงหน้าก่อนที่จะมีคนกด Export

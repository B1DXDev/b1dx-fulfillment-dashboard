สำหรับ B1DX

Login ครั้งเดียว เข้าได้ทุกระบบ — ปลอดภัย ตรวจสอบได้ ขยายได้
วันที่จัดทำ มิถุนายน 2026 สถานะ ⚠️ Decision Reversed — ดู update ด้านล่าง ครอบคลุม 8 ระบบ
⚠️ อัปเดต 2026-06-18 — มติเปลี่ยนแล้ว

Brief นี้เขียนตอน ADR-005 ยังไม่ build จริง (มิถุนายน 2026 ต้น) — หลังประเมินเชิงลึกเพิ่มเติม ทีมสถาปัตยกรรมกลับมติ จาก "สร้าง OIDC server เอง (OpenIddict)" มาเป็น รับ Keycloak (self-hosted) เป็น Central IdP แทน ดู ADR-009 (Accepted, supersedes ADR-005) และ SSO Developer Guide สำหรับสถาปัตยกรรมที่ build จริง — เนื้อหาส่วน "ทำไมสร้างเอง ไม่ซื้อ Keycloak" ด้านล่างเป็นเหตุผลเดิมตอนตัดสินใจครั้งแรก เก็บไว้เป็น historical record เท่านั้น

User Experience — พนักงานใช้งานอย่างไร
ก่อน SSO (ปัจจุบัน)
  • Login แยกทุกระบบ
  • จำหลายรหัสผ่าน
  • เปลี่ยนรหัสต้องทำทุกระบบ
  • Admin ยกเลิกสิทธิ์ต้องทำทีละระบบ
  • ไม่มี Audit Log รวม
  • ไม่มี Social Login
หลัง SSO (ใหม่)
  • Login ครั้งเดียว เข้าได้ทุกระบบ
  • บัญชีเดียว รหัสเดียว (หรือ Social)
  • เปลี่ยนรหัสที่เดียว มีผลทุกระบบ
  • ยกเลิกสิทธิ์พนักงานที่เดียว ตัดทุกระบบ
  • Audit Log รวมศูนย์ ตรวจสอบง่าย
  • Google / Microsoft / LINE / etc.

จุดสำคัญ: Phase 2 ใช้ "Dual-Token" — OMS รับทั้ง Token เดิมและ Token ใหม่ได้พร้อมกัน ทำให้ไม่มี Downtime ระหว่าง Migrate. Password เดิมของ User ทุกคนไม่ต้อง Reset
📜 ทำไมสร้างเอง ไม่ซื้อ Keycloak / Auth0 / Okta? (เหตุผลเดิม — ถูก reverse แล้ว ดู ADR-009)
ถ้าใช้ Keycloak / Auth0 / Okta
  • Password Hash ไม่เข้ากัน — ต้อง Force Reset ทุก User
  • Auth0 คิดค่าบริการตาม MAU (Monthly Active Users)
  • ต้องเรียนรู้ Stack ใหม่ ทีมไม่ถนัด
  • Data อยู่กับ Vendor — ควบคุมได้น้อยกว่า
  • ปรับแต่ง Multi-Tenant Model เดิมยาก
ทำไม B1DX Identity Platform ดีกว่า
  • Password Hash เดิมใช้ได้เลย — ไม่ต้อง Reset
  • ไม่มีค่า License รายเดือน
  • ทีมถนัด .NET อยู่แล้ว (~70% reuse โค้ดเดิม)
  • Data อยู่ใน Control เรา 100%
  • Multi-Tenant Model ที่มีอยู่แล้วใช้ต่อได้

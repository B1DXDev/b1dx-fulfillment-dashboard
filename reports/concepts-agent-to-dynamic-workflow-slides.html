LEVEL 0 · พื้นฐานที่สุด

เริ่มจาก LLM เปล่า ๆ

โมเดลภาษา (เช่น Claude) รับข้อความ → ตอบข้อความ. ฉลาดแต่ "ทำอะไรเองไม่ได้" — อ่านไฟล์ไม่ได้ รันคำสั่งไม่ได้ จำรอบที่แล้วไม่ได้

prompt → [ LLM ] → text
ข้อจำกัด: เป็นแค่ "สมอง" ที่ไม่มีมือ ไม่มีความจำ ไม่มีเป้าหมายต่อเนื่อง — ตอบจบเป็นครั้ง ๆ
LEVEL 25 · Agent คืออะไร

Agent = LLM ที่ "ลงมือทำ" ได้

เอา LLM มาใส่ 4 อย่าง แล้วให้มันวน loop จนบรรลุเป้าหมาย — นี่คือ agent

🎯

Goal

เป้าหมายที่ต้องทำให้สำเร็จ ไม่ใช่แค่ตอบคำถาม

🛠️

Tools

อ่าน/เขียนไฟล์, รันคำสั่ง, เรียก API — "มือ" ของมัน

🧠

Context/Memory

รู้สถานะปัจจุบัน + บทบาทของตัวเอง

🔁

Loop

ทำ → ดูผล → ตัดสินใจ step ถัดไป จนเสร็จ

LLM + Tools + Goal + Loop = Agent

LEVEL 50 · Agentic คืออะไร

"Agentic" เป็น คุณสมบัติ ไม่ใช่ของชิ้นหนึ่ง

Agent = คำนาม (ตัวตน) · Agentic = คำคุณศัพท์ (ระดับความเป็นตัวของตัวเอง). ยิ่ง agentic = ยิ่งตัดสินใจเองได้มาก ไม่ต้องสั่งทีละ step

🧭

วางแผนเอง

แตกงานใหญ่เป็น step ย่อยได้เอง

🔀

ตัดสินใจเอง

เลือกว่าจะทำอะไรต่อจากผลลัพธ์ที่เห็น

🩹

แก้ทางเอง

เจอ error → หา root cause → ลองใหม่

สเกล: สั่งทีละบรรทัด (ไม่ agentic) ⟶ บอกเป้าหมายแล้วมันจัดการเอง (agentic เต็มที่). มันเป็น "เข็มวัด" ไม่ใช่ on/off
เคลียร์จุดสับสนที่ 1

Agent ≠ Agentic

AgentAgentic
ชนิดคำคำนาม — "ตัวตน"คำคุณศัพท์ — "คุณสมบัติ"
ตอบคำถามใคร/อะไรทำงานทำงานแบบอิสระแค่ไหน
วัดยังไงมี/ไม่มีมาก/น้อย (สเกล)
ตัวอย่าง"be-coder เป็น agent""flow นี้ agentic เพราะเลือกทางเอง"
พูดได้ว่า "agent ที่ทำตัว agentic" — agent คือตัว, agentic คือพฤติกรรมของมัน
LEVEL 75 · Agentic Workflow คืออะไร

ร้อย agent หลายตัว + ให้ระบบตัดสินใจ

เมื่องานใหญ่เกินกว่า agent ตัวเดียว — เราเอาหลายตัวมาต่อกัน โดยมี "ตัวประสาน" (orchestrator) ที่ตัดสินใจว่าใครทำอะไรต่อ

orchestrator→ architect→ spec-writer→ be ∥ fe→ reviewer
เส้นแบ่งสำคัญ: ถ้าตัวประสาน "ตัดสินใจด้วย LLM" (เลือกทาง, ประเมินว่าพอหรือยัง, แก้เมื่อ fail) = agentic workflow. ถ้าเดินตาม if-else ตายตัว = แค่ scripted/pipeline

Scripted Workflow vs Agentic Workflow

มิติScripted (pipeline)Agentic
เส้นทางตายตัว เขียน hardcodeตัดสินด้วย LLM ตาม context
เจอเคสใหม่ต้องแก้โค้ด flowปรับตัวได้เอง
"พอหรือยัง"จบตาม stepประเมินเอง / loop จน done
b1dx ตอนนี้← อยู่ตรงนี้ (3-phase non-negotiable)← เป้าหมาย

LEVEL 100 · Dynamic Workflow คืออะไร

เส้นทางเปลี่ยนตามชนิดงาน

รูปแบบที่ชัดที่สุดของ agentic workflow: classify ก่อน → เลือก route → ข้าม step ที่ไม่ต้อง. งานคนละแบบเดินคนละเส้น

bug-fix classify→ architectspec→ regression test→ fix→verify
feature/HIGH classify→ architect→spec→ 🛡 security→ be ∥ fe→ reviewer
Dynamic = "งานเลือกเส้นทางของตัวเอง" — bug เล็กข้าม design, OAuth บังคับ security. นี่คือสิ่งที่ b1dx กำลังจะ refactor ไปหา
เคลียร์จุดสับสนที่ 3 (ถามบ่อยสุด)

"Agentic" ต่างจาก "Multi-agent" ยังไง

คนละแกนกันคนละเรื่อง — ไม่ใช่ของคู่แข่ง

Multi-agentAgentic
แกนที่พูดถึงจำนวน agent (มากกว่า 1)ระดับความอิสระในการตัดสินใจ
คำถาม"มีกี่ตัว?""ตัดสินใจเองแค่ไหน?"
ตรงข้ามคือSingle-agent (ตัวเดียว)Scripted (ตายตัว)
เป็นอิสระต่อกัน✓ มี multi-agent ที่ scripted ก็ได้ · มี single-agent ที่ agentic ก็ได้
b1dx = multi-agent (10 ตัว) + กำลังเพิ่มความ agentic — เรามีหลายตัวอยู่แล้ว สิ่งที่เพิ่มคือให้ระบบ "เลือกเส้นทางเอง"
มองเป็น 2 แกน

จำนวน × ความอิสระ = 4 แบบ

Single + Scripted

agent ตัวเดียว ทำตาม step ตายตัว · เช่น สคริปต์ automation ธรรมดา

Single + Agentic

agent ตัวเดียวแต่ตัดสินใจเอง · เช่น Claude Code แก้ bug ตัวเดียวจบ

Multi + Scripted

หลายตัวแต่เดินท่อตายตัว · ← b1dx ตอนนี้

Multi + Agentic

หลายตัว + เลือกเส้นทางเอง · ← เป้าหมาย (dynamic workflow)

สรุปให้จำ

5 ประโยคที่อยากให้ทีมจำกลับไป

  • Agent = LLM + tools + goal + loop → "ตัวที่ลงมือทำ"
  • Agentic = คุณสมบัติ "ตัดสินใจเองได้มากแค่ไหน" (สเกล ไม่ใช่ on/off)
  • Agentic Workflow = ร้อย agent + ให้ระบบตัดสินใจเส้นทาง (ไม่ใช่ if-else)
  • Dynamic Workflow = agentic workflow ที่เลือก route ตามชนิดงาน/risk
  • Agentic ≠ Multi-agent — คนละแกน (ความอิสระ vs จำนวน); b1dx มี multi อยู่แล้ว กำลังเพิ่ม agentic

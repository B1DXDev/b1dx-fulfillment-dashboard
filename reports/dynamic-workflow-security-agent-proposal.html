ข้อเสนอ — Dynamic Workflow

เพิ่ม Phase 0 (Classify) + Route Table — งานต่างประเภทเดินคนละเส้น

หลักการ

Orchestrator classify ก่อน ว่างานคือ type อะไร + risk เท่าไหร่ → ประกาศ route ที่เลือก → ค่อย dispatch. แต่ละ type ข้าม phase ที่ไม่จำเป็นได้ (มี accountability ผ่าน plan.md)

เพิ่ม Security Agent

1 agent · 4 modes — เติม gap ที่ Reviewer §8 (PR-time) ครอบไม่ถึง
💡
Recommendation: สร้าง 1 agent (agents/security.md) ที่มี 4 mode — ไม่ใช่ 4 agents แยก (กัน context switch + ซ้ำ CLAUDE.md ต่อ repo) คล้าย UI Tester ที่มี scaffold + run. เริ่ม threat mode ก่อน — ROI สูงสุด

4 Modes

ModePhaseทำอะไรTrigger
threat1STRIDE / attack surface ก่อนเขียนโค้ดspec แตะ auth/tenant/PII/external API
scan2อ่าน SAST/secret/dep report → block PR ไหมทุก build (∥ coders)
pentest3DAST + IDOR + auth bypass testpayment/OAuth/file-upload/public endpoint
hardeningQuarterlyTLS / secret rotation / WAF / RBAC baselinesprint-end / new repo / new integration

Gap ที่ปิด (เทียบ 4-layer)

Layerตอนนี้หลังเพิ่ม
L1 — Threat modelingไม่มีthreat mode
L2 — SAST / secret / dep scanไม่มีscan mode + CI
L3 — Penetration testบางส่วน (Reviewer §8)pentest mode
L4 — Infra hardeningไม่มีhardening mode
⚠️
ห้าม duplicate Reviewer §8 (OWASP/tenant/JWT/secrets มีอยู่แล้ว). Security agent เติม สิ่งที่ Reviewer ทำไม่ได้เพราะมาช้า (PR-time) — โดยเฉพาะ pre-design + scan automation + post-deploy

ภาพรวม Flow ใหม่

Security เป็น conditional node ที่ถูก trigger โดย risk — ไม่ใช่ทุก feature (กัน bottleneck)
feature (high-risk) classify→ architect→ spec→ security:threat→ BE∥FE+ security:scan→ ui-tester→ security:pentest→ reviewer
bug-fix classify→ architect→ spec→ regression test→ fix→ verify
chore/docs classify→ all test gates→ coder→ verify build
base ขั้นปกติ security conditional (risk) cond optional skip ข้ามตาม route

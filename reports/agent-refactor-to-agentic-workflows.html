Refactor Agents → Agentic Workflows

● ปัจจุบัน — Procedural

Orchestrator เป็น "สคริปต์ในหัว"

logic การ route ฝังเป็น prose ใน orchestrator.md §5.2/§17 · dispatch ผ่าน prose template · ทุก request เดิน Phase 1→1.5→2→3 เหมือนกันหมด · handoff = ก๊อปข้อความยาว

● เป้าหมาย — Agentic

Workflow เป็น "ผังที่ประกาศได้"

classify เป็น step แรก · route table แยกไฟล์ · งานต่างชนิดเดินคนละเส้น · handoff = structured contract · route ที่เลือกถูกประกาศ + บันทึกใน ledger

1

4 การเปลี่ยนหลัก (Paradigm Shift)

นิยามของ "agentic workflow" คือ 4 คุณสมบัตินี้ที่ procedural pipeline ไม่มี
มิติปัจจุบัน (Procedural)หลัง Refactor (Agentic)
① Routing Implicit — orchestrator อ่าน prose แล้วตัดสินใจในหัว ทุกงานมักลง full pipeline Declarative — workflow-router.md มี route table; classify → เลือกเส้น → ข้าม phase ที่ไม่ต้อง
② Handoff Prose dispatch template (orchestrator §10) — ก๊อปข้อความยาว เสี่ยงตกหล่น Structured contract — input/output schema ต่อ stage (spec path, branch, TC IDs, done-gate) ตรวจครบก่อนส่ง
③ Control flow Fixed sequence — "3 Phases Non-Negotiable" Conditional + parallel — node รัน/ข้าม ตาม type · security เป็น conditional node ตาม risk
④ Observability รู้ว่าเกิดอะไรจาก chat log เท่านั้น Announced + logged — orchestrator ประกาศ route ที่เลือก + เขียน Routing section ใน plan.md
🔑
สรุปแก่น: agentic workflow = "งานเลือกเส้นทางของตัวเองตาม context" ไม่ใช่ "ทุกงานเดินท่อเดียว". เรามีวัตถุดิบครบแล้ว (10 agents · plan.md state · stop list) — แค่ยก routing จาก prose ขึ้นเป็น declarative step
2

เห็นภาพ: เส้นทางงานเดียวกัน

ตัวอย่าง — งาน "แก้ bug ปุ่มไม่ disable" (FE bug-fix เล็ก)
✗ ปัจจุบัน — เดิน full pipeline
1. Phase 1 — architect/spec design
2. Phase 1.5 — ui-tester scaffold
3. Phase 2 — dispatch BE ∥ FE
4. Phase 3 — verify ทุก repo
5. ui-tester run + reviewer

→ ลาก architect, spec-writer, BE coder, ui-tester ทั้งที่งานแตะแค่ 1 ไฟล์ FE — เปลือง token + ช้า

✓ หลัง refactor — route: bug-fix
0. classify → bug-fix / low risk
— architect (skip)
— spec-writer (skip)
1. regression test (reproduce)
2. FE coder fix
3. verify (lint+typecheck+e2e)

→ 3 ขั้น แทน 5 phase · เรียกแค่ 2 agent · accountability ยังอยู่ครบใน plan.md

ตัวอย่าง — งาน "เพิ่ม OAuth login marketplace" (feature เสี่ยงสูง)
✗ ปัจจุบัน — security เป็น optional
1. Phase 1 — design
2. Phase 1.5 — scaffold
3. Phase 2 — BE ∥ FE
4. Phase 3 — verify
5. reviewer (ถ้านึกออกว่าควรเรียก)

→ threat modeling ไม่มี · security ขึ้นกับว่า orchestrator "นึกได้" ไหม

✓ หลัง refactor — route + risk override
0. classify → feature / HIGH (OAuth)
1. architect → spec
🛡 security:threat (บังคับ)
2. BE ∥ FE + 🛡 security:scan
3. ui-tester → 🛡 security:pentest
4. reviewer (opus, บังคับ)

→ risk override บังคับ security เข้า flow อัตโนมัติ — ไม่พึ่งความจำคน

ขั้นที่รัน ข้ามตาม route security node (conditional)
3

ผลกระทบต่อ agent แต่ละตัว

ข่าวดี — agent body แทบไม่ต้องแก้ ส่วนใหญ่แค่ "ถูกเรียกแบบมีเงื่อนไข"
Agentเปลี่ยนอะไรสถานะ
orchestratorเพิ่ม Phase 0 (classify) · §4 จาก "non-negotiable" → "route กำหนด phase" · อ้าง routerแก้หลัก
architectbody เดิม — เปลี่ยนเป็น conditional (เรียกเฉพาะ feature/migration/greenfield)คงเดิม
spec-writerbody เดิม — skip ใน bug-fix/chore routeคงเดิม
be / fe / sync / webhook coderbody เดิม — รับ structured handoff แทน prose (ดีขึ้น, ไม่ตกหล่น)คงเดิม
ui-testerbody เดิม (มี 2-mode อยู่แล้ว) — skip ใน chore/spike routeคงเดิม
reviewerbody เดิม — risk override บังคับเรียก (opus) เมื่อแตะ sensitive surface+ trigger rule
sprint-plannerไม่เกี่ยวกับ build route — คงเดิมคงเดิม
🆕 securityagent ใหม่ 4-mode (เริ่ม threat) — conditional node ตาม riskเพิ่มใหม่
✅
8 / 10 agent body ไม่ต้องแตะ. การ refactor นี้คือเรื่องของ orchestration layer (วิธีร้อย agent) — ไม่ใช่ rewrite ตัว agent. ความเสี่ยงต่ำ, rollback ง่าย (ลบ router + คืน §4 เดิม)

5

แผน Migration — 3 สัปดาห์ (incremental)

ไม่ big-bang — ของเดิมยังใช้ได้ตลอด ทยอยเปิด route ทีละชนิด
Week 1Foundation
  • สร้าง workflow-router.md + route table (feature / bug-fix / chore ก่อน)
  • เพิ่ม Phase 0 classify ใน orchestrator (ยังไม่บังคับ skip — แค่ประกาศ route)
  • ทดลองกับงานจริง 2-3 ใบ → ดูว่า classify ตรงไหม
Week 2Dynamic routing
  • เปิด skip จริงต่อ route (bug-fix ข้าม architect/spec ได้)
  • เพิ่ม /lead command + Routing section ใน plan.md
  • เพิ่ม route: ui-only / migration / sync / webhook / spike
Week 3Security node
  • สร้าง agents/security.md (threat mode)
  • ผูก risk override → inject security:threat + reviewer บังคับ
  • เพิ่ม Test Type Security + Security Review ใน plan template
⚠️
กฎความปลอดภัยของ migration: Hard Gates เดิม (branch discipline, commit boundary, verify gates, plan.md lifecycle) ห้ามแตะ — เปลี่ยนแค่ "เรียก agent ตัวไหนเมื่อไหร่" ไม่เปลี่ยน "อะไรที่บังคับก่อน commit"

ขอ Decision จากทีม

  • เห็นด้วยกับแนวทาง incremental 3 สัปดาห์ หรืออยากเริ่มแค่ route bug-fix ก่อนพิสูจน์?
  • classification ใครเป็นคนตัดสิน — orchestrator auto หรือ dev เลือก type ตอนเปิดงาน?
  • route ชุดแรก (feature/bug-fix/chore) พอไหม หรือต้องมี migration ตั้งแต่ต้น?
  • security node — รวมในเฟสนี้ หรือแยกเป็น proposal ต่างหาก (มีเอกสารแยกแล้ว)?

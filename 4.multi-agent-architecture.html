2 แยก Repo = แยก Context = แยก Agent
ไม่ต้องกลัว project บวม · 10 หรือ 20 repo ก็ทำงานเร็วเหมือนเดิม เพราะ context ของ agent ถูก scope ด้วย repo
FIG. 3.1 — Multi-Repo Architecture · แต่ละ agent ทำงาน scope เฉพาะ repo ของตัวเอง สื่อสารผ่าน contract ที่ workspace
6 ข้อดี ของ multi-repo workspace
1
แยกอิสระ
แต่ละ repo มี git history · branch · CI/CD ของตัวเอง — ไม่กระทบกันเมื่อ commit หรือ deploy
2
Context ไม่บวม
Agent โหลดแค่ repo ที่ตัวเองทำงาน — ไม่ต้องอ่านทุก codebase token ต่ำลง ตอบเร็วขึ้น
3
Agent เฉพาะทาง
backend-coder รู้ .NET / EF Core ลึก · webhook-coder รู้ Go / Gin ลึกกว่า agent รวมทุกอย่าง
4
ทำงานขนาน
Backend agent แก้ API ขณะ Frontend agent ทำ UI พร้อมกัน — ไม่ block กันเพราะคนละ repo
5
เพิ่ม repo ใหม่ง่าย
เพิ่ม submodule + agent config เสร็จในไม่กี่นาที — 10–20 repo ก็ไม่บวม
6
Deploy / Scale แยก
แต่ละ repo มี pipeline ของตัวเอง — rollback / scale up เฉพาะส่วนได้ ไม่กระทบกัน
เปรียบเทียบ — Monolith vs Workspace
Monolith โตขึ้น → ช้าลง · agent ค้าง
  • Repo เดียวรวมทุกอย่าง — agent ต้อง index ทั้ง project
  • เพิ่ม feature → context window พุ่ง → agent ช้าและตอบงงๆ
  • Build / Deploy ผูกกัน — เปลี่ยน FE ต้อง re-deploy BE
  • 1 agent ค้าง → agent อื่นรอ lock เดียวกัน
Workspace โตขึ้น → แค่เพิ่ม repo ใหม่
  • แต่ละ repo เป็น scope เฉพาะ — context window ยังเล็ก เร็วเสมอ
  • เพิ่ม feature → spawn agent ใหม่ใน repo ใหม่ ไม่กระทบของเดิม
  • Build / Deploy แยก — เปลี่ยน FE ไม่ต้องแตะ BE
  • Agent ทำงานขนานในคนละ repo — ไม่มี lock กลาง
แก่นความคิด: "แยก codebase = แยก context = แยก agent ความเชี่ยวชาญ" เมื่อ context ของแต่ละ agent ถูกจำกัดด้วย repo เดียว — agent ไม่ต้องรู้ทุกอย่างของระบบ ไม่ต้องเหนื่อยกับการ index/อ่านไฟล์ที่ไม่เกี่ยว ผลคือ คุณภาพการทำงานดีขึ้นตามจำนวน repo ไม่ใช่แย่ลง
3 Key Principles
หลักการที่ทำให้ระบบยืดหยุ่นและ scale ได้
1
Workspace ไม่เก็บ source code

Workspace เป็น orchestration layer สำหรับรวม context (specs, mockups, agent prompts, task tracking) ไม่มี code จริงอยู่ใน ./be-repo/, ./web-repo/ — ใช้แค่เป็น submodule reference

2
แต่ละ agent มี repo อิสระ

Git history, branch, deployment pipeline แยกชัดเจน — FE Agent แก้ UI ก็ commit เฉพาะใน b1dx-oms-fulfillment-web เท่านั้น ไม่กระทบ repo อื่น

3
Contract-based communication

Agent คุยกันผ่าน contract ที่ Workspace ประกาศไว้ (API spec, event schema, shared types) — ไม่มี direct coupling ระหว่าง repo

4
เปลี่ยน 1 repo ไม่กระทบ repo อื่น

ถ้า contract ยังคงเดิม → deploy แยกได้, rollback แยกได้, ทำ A/B test ได้ — ต้อง bump version contract เมื่อ breaking change เท่านั้น

5
Orchestrator coordinates · ไม่ execute เอง

Orchestrator (claude ที่อ่าน agents/orchestrator.md) ตัดสินใจว่างานไหนส่งให้ agent ไหน · spawn parallel ได้ · รวมผลลัพธ์กลับมา · ห้าม agent ตัวเดียวทำงานข้าม repo

5 Runtime Flow
คำสั่ง user ไหลผ่าน Orchestrator → Agents → Repos → กลับมา user อย่างไร
1
USERส่งคำสั่ง
เช่น build feature order-detail-improve หรือ fix bug FE-EOD-04
2
ORCHESTRATORอ่าน spec + วิเคราะห์ scope
โหลด docs/specs/{feature}.md, mockup, อ่าน tasks/current-sprint.md → ตัดสินใจว่ากระทบ repo ใดบ้าง (BE? FE? Sync? Webhook?)
3
ORCHESTRATORนำเสนอ plan + รอ user trigger
แสดง 4 รายการ (scope · summary of changes · SIT impact · test plan) → สร้าง tasks/plans/{TICKET}/plan.md → รอ user พิมพ์ build หรือ ลุย
4
ORCHESTRATORspawn subagents แบบ parallel
spawn BE Coder + FE Coder พร้อมกันใน working dir ของแต่ละ repo · UI Tester scaffold test ก่อน (Phase 1.5) · Sync/Webhook spawn ถ้า scope เกี่ยว
5
AGENTทำงานใน repo ของตัวเอง
แต่ละ agent: git checkout -b {branch} → TDD (test ก่อน impl ทุก layer) → verify (dotnet test / pnpm test / go test) → stage files
6
ORCHESTRATORรวมผลลัพธ์
รับ report กลับจากทุก agent · spawn UI Tester อีกครั้งเพื่อ un-skip + รัน e2e · update plan.md Status → In Review
7
USERสั่ง commit / push / PR
Agent ห้าม auto-push — รอ user พิมพ์ commit / push / PR แล้วจึงรันคำสั่งจริง · ห้ามตรงเข้า main เด็ดขาด
6 Development Workflow
ใครแก้อะไร · branch / commit / merge กันยังไง · contract sync
Branch Convention
ทุก agent ต้อง:
  • git checkout main && git pull
  • git checkout -b feat/{ticket} หรือ fix/{ticket}
  • ห้าม commit ลง main ตรง — push branch → gh pr create
Plan Gate (ทุก task)
ก่อนแก้โค้ดต้อง:
  • แสดง 4 รายการใน chat (scope · changes · SIT impact · test plan)
  • สร้าง tasks/plans/{TICKET}/plan.md
  • รอ user พิมพ์ build หรือ ลุย
  • หลัง trigger → build ต่อเนื่อง ไม่ confirm กลางทาง
Parallel Build
BE + FE spawn พร้อมกันได้ทันที:
  • ถ้า BE ยังไม่พร้อม → FE ทำ MSW mock ก่อน (ไม่รอ)
  • ทำงานคนละ repo จึงไม่เกิด git conflict
  • ห้าม spawn parallel subagent บน repo เดียวกัน
Contract Sync
API spec / event schema = source of truth ที่ Workspace:
  • BE ประกาศ contract ใน docs/specs/{feature}.md
  • FE / Webhook / Sync อ่าน spec เดียวกัน — ห้าม copy
  • Breaking change → bump version + แจ้ง consumer ทุกตัว
Task Assignment (3 ไฟล์)
ทุก task ต้อง update พร้อมกัน:
  • tasks/backlog.md § Issues Backlog
  • tasks/current-sprint.md (effort, AC, branch)
  • tasks/assignments/YYYY-MM-{name}.md § Active
Test Discipline
ห้าม mark done ถ้าไม่มี test:
  • FE: e2e/{feature}/{feature}.spec.ts ผ่านครบ
  • BE: unit + integration ครอบคลุม happy + error
  • Sync: unit หรือ TestContainers integration
  • Visual fidelity: screenshot vs mockup (FE)
PR Coordination
แต่ละ repo มี PR แยก:
  • BE PR merge ก่อน → FE PR consume API จริง
  • FE PR ตอน BE ยังไม่ merge → mock ด้วย MSW
  • Reviewer ต้องเปิด mockup จริง + ดู screenshot ก่อน approve
Post-Push Hygiene
หลัง push ทุกครั้ง:
  • รัน node scripts/gen-progress.mjs
  • Update plan.md Status → In Review (PR link)
  • Merged → Update Status → Done + date
7 Contract Example
ตัวอย่างจริง: feature "Order Created" — agent คุยกันผ่าน contract อย่างไร
BE 1. ประกาศ endpoint ใน spec
BE Agent เขียน docs/specs/order-create.md ระบุ contract → consumer ทุกตัวอ่านจากที่นี่
# API Contract — POST /api/orders Request: tenant_id: # ห้ามรับจาก body — ดึงจาก ICurrentTenant channel: "shopee" | "lazada" | "tiktok" items[]: sku: string qty: int price: decimal(18,2) Response 201: order_id: uuid status: "pending" created_at: iso8601 Emits event: order.created # → RabbitMQ exchange
↓
FE 2. Frontend consume API
FE Agent สร้าง zod schema + hook จาก contract เดียวกัน → ถ้า BE ยังไม่พร้อม ใช้ MSW mock ตอบตาม contract
// features/orders/schemas/create-order.ts export const CreateOrderSchema = z.object({ channel: z.enum(["shopee", "lazada", "tiktok"]), items: z.array(z.object({ sku: z.string(), qty: z.number().int().positive(), price: z.number().positive(), })), }); // hooks/use-create-order.ts export const useCreateOrder = () => useMutation({ mutationFn: (data) => api.post("/api/orders", data), });
↓
WEBHOOK 3. Webhook รับจาก marketplace → call BE API
Webhook Agent รับ event จาก Shopee → return HTTP 202 ทันที → publish to RabbitMQ → worker consume → call BE API ตาม contract เดียวกัน
// internal/handler/shopee_order.go func (h *Handler) ShopeeOrderCreated(c *gin.Context) { payload := parseShopeePayload(c) h.queue.Publish("order.shopee.created", payload) // idempotent key c.JSON(202, gin.H{"status": "accepted"}) } // internal/service/order_consumer.go (worker) func (s *Service) Consume(msg Message) error { req := mapShopeeToOmsContract(msg) // → ตาม spec ของ BE return s.omsClient.CreateOrder(req) }
↓
SYNC 4. Sync Worker subscribe event
Sync Worker Agent subscribe order.created event → push ไป ERP (B1) — ใช้ event schema เดียวกับที่ BE ประกาศ
// Application/Handlers/OrderCreatedHandler.cs public class OrderCreatedHandler : IConsumer<OrderCreatedEvent> { public async Task Consume(ConsumeContext<OrderCreatedEvent> ctx) { var sapOrder = MapToSapOrder(ctx.Message); // schema จาก spec await _b1Client.PushOrderAsync(sapOrder, ctx.CancellationToken); } }
Key insight: 4 agent / 4 repo / 4 ภาษา (C# / TypeScript / Go / C#) แต่ทำงานร่วมกันได้เพราะ contract เดียวกันที่ Workspace — ถ้า BE bump version contract → ทุก consumer (FE, Webhook, Sync) ต้อง update tests → reviewer block PR ถ้าไม่ครบ
